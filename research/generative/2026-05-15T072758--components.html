ARA Research Component Reference

Every primitive available to a research article, used at least once. If you're reading this, the design system works end-to-end.

Articles in the Research tab compose from a fixed vocabulary of ara-* classes. The writer parses every fragment at commit time and rejects any class outside the allowlist or any HTML tag outside the semantic whitelist. The dashboard ships the matching CSS; edit one file to restyle every article.

This document is its own integration test. It exercises every primitive at least once. If a primitive renders wrong here, it renders wrong everywhere.

01Headings and structure

Use ara-display for the article title (one per doc). Use ara-h2 for sections (with an optional ara-h2-num chip for numbered sections). ara-h3 and ara-h4 step down from there.

An ara-h3 subsection

Subsections sit beneath h2-level sections. Lighter weight, smaller size.

An ara-h4 minor head

Minor heads are for tight subdivisions inside a subsection. Same family, smaller and muted.

02Callouts

Four variants. Use sparingly — every callout costs reader attention.

Note

Informational callouts use the warm accent color. Good for footnotes, asides, methodology disclosures.

Confirmed

Green for verified facts, completed actions, positive signals. Use when something is genuinely settled, not just nice.

Caveat

Yellow for partial confidence, unverified claims, things to watch. Most analytical hedges land here.

Risk

Red for red flags. Don't dilute by using it for mere uncertainty — that's what warn is for.

03Pull quote

For a single sentence worth slowing down for. Don't use for normal citations — that's what links are for.

The bottleneck is no longer compute. It's the human attention required to know what to compute on. Composite take, AI research bench, mid-2026

07Lists, links, inline elements

Standard unordered list:

  • A claim with an inline source.
  • Another claim with emphasis, strong emphasis, and inline code.
  • A statement followed by a numbered citation1.

Ordered list:

  1. First step in a sequence.
  2. Second step.
  3. Third step.

08Tags, flags, highlights

Inline tags: cdn security edge-compute

Status flags inline: solid; watch; concerning.

Highlighted text: key insight draws the eye.

09Bars — single-value comparison

For when "$520M" in a stat block isn't enough — you want the reader to see the proportion. Each ara-bar declares its fill via data-pct; the dashboard sets the CSS variable at render time so authors don't touch inline style.

Security $520M · 62%
Delivery $380M · 45%
Compute $180M · 22%

10Stacked bar — segmented proportions

One horizontal bar split by category, with a legend underneath. Four neutral fill shades (ara-stack-seg--1 through --4) keep it monochromatic.

  • Security 42%
  • Delivery 28%
  • Compute 18%
  • Other 12%

For ratio-at-a-glance comparisons: pick a glyph, set data-count, and the dashboard expands it into repeated child spans at render time. Cap is 200. Pair two rows for "then vs. now."

15Inline accent + unit suffix

Use ara-accent to anchor the eye on "the subject" across a global comparison — distinct from <em> (italic) and ara-mark (background tint). Use ara-stat-unit for tiny unit suffixes that hang off big figures like 87 yrs. or $39.6T.

16Code blocks

Inline code and fenced blocks:

def compose(article: str) -> bool:
    # Every class must start with ara-
    return all(c.startswith("ara-") for c in classes_in(article))

17What's NOT here

Components that don't exist by design:

  • No layout primitives. No ara-grid, ara-row, ara-col. Articles flow top-to-bottom; if you need a grid, it's probably ara-stats or a table.
  • No color utilities. No ara-text-red. Use a callout or a flag for semantic color.
  • No spacing utilities. Margins are baked into each component. If two components feel cramped, the component's own margins should be edited.
  • No theming knobs. Light/dark come from the dashboard's CSS variables. Articles never pick colors directly.
