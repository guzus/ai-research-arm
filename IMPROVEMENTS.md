## Issues Found in 2026-05-01 Output

### 1. Bluesky pipeline has been emitting empty digests for 5 days running

`research/bluesky/2026-05-01.md`:

> _No posts available for this run: upstream Bluesky fetch returned
> HTTP 403 Forbidden for all source files (ai_announcements,
> llm_releases, model_discussions, ml_papers, karpathy). Skipping
> curation; next scheduled run will retry._

Same string appears in `2026-04-30.md`, `2026-04-29.md`, `2026-04-28.md`,
and earlier days. Bluesky has effectively been a dead source since
~2026-04-27.

**Root cause confirmed by curl probe from this runner:**

```
$ curl -i https://public.api.bsky.app/xrpc/app.bsky.feed.searchPosts?q=AI&limit=3
HTTP/2 403
... <html>403 Forbidden ...
```

The CDN/anti-bot layer in front of `public.api.bsky.app` started rejecting
unauthenticated XRPC calls from datacenter / GitHub-runner IP ranges.

**Fix verified:** the alternate host `api.bsky.app` (no `public.` prefix)
returns HTTP 200 with valid JSON for the same requests:

```
$ curl -w "%{http_code}\n" "https://api.bsky.app/xrpc/app.bsky.feed.searchPosts?q=AI&limit=3"
200
{"posts":[{"uri":"at://did:plc:msian4dqa2rqalf3biilnf3m/...

$ curl -w "%{http_code}\n" "https://api.bsky.app/xrpc/app.bsky.feed.getAuthorFeed?actor=karpathy.bsky.social&limit=2"
200
{"feed":[{"post":{...}}, ...]}
```

Same fix was proposed in `improve/2026-04-29` (still-open PR) but
was bundled in a `git am`–style patch the maintainer hasn't landed.
Re-issuing today as a clean standalone `git apply` patch with a
narrower diff and additional hardening (see Changes Made).

### 2. The previous `from:` searchPosts trick never worked

The current workflow tries to capture Karpathy's posts via:

```
curl ".../searchPosts?q=from:karpathy.bsky.social"
```

But Bluesky's search XRPC does **not** implement a `from:` operator —
this query returns `{"posts":[]}` even when `public.api.bsky.app`
isn't returning 403. So even before the 403 regression, this file
contributed zero data to the digest. The right primitive is
`app.bsky.feed.getAuthorFeed?actor=<handle>`.

### 3. Researcher coverage is limited to one handle

Even once the 403 is fixed, only one researcher's feed is captured
(Karpathy). The Bluesky AI-research community has grown — Sebastian
Raschka, Simon Willison, Ethan Mollick, and Sam Altman all post
substantively there now and aren't reflected in the digest.

### `.github/workflows/2h-bluesky.yml` (in patch)

1. **Host swap**: `public.api.bsky.app` → `api.bsky.app` for all
   XRPC calls. Restores HTTP 200 JSON responses (verified by direct
   curl probe).

2. **Replace broken `from:` trick with proper `getAuthorFeed`** for
   five high-signal AI researchers:

   | Handle                       | Who                |
   |------------------------------|--------------------|
   | `karpathy.bsky.social`       | Andrej Karpathy    |
   | `rasbt.bsky.social`          | Sebastian Raschka  |
   | `simonwillison.net`          | Simon Willison     |
   | `emollick.bsky.social`       | Ethan Mollick      |
   | `sama.bsky.social`           | Sam Altman         |

   `getAuthorFeed` returns posts under `feed[].post.*` instead of
   `posts[].*`, so the Claude prompt is updated to handle both
   shapes (with explicit "if file missing or empty, skip — do NOT
   fail the run" guardrail).

3. **HTML-response normalization**: any source file whose first
   byte is `<` (i.e. an HTML error page snuck through) is rewritten
   to `{"posts":[]}` or `{"feed":[]}` before the Claude step runs.
   Prevents future CDN regressions from crashing the prompt with
   HTML-as-JSON.

4. **`curl --retry 3 --retry-delay 2`** on every fetch. Transient
   Bluesky 5xx now retry instead of silently producing empty
   source files.

5. **First-byte audit logged** at the end of the fetch step so any
   future regression is visible in the runner log without having
   to inspect the JSON files.
