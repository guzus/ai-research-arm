## Why this keeps recurring

Today is the **sixth consecutive `daily-improve` PR** to identify
the same two silently-broken feeds:

- `improve/2026-05-12`, `2026-05-13`, `2026-05-14`, `2026-05-16`, today —
  all carry effectively the same diagnosis and the same fix.
- The root blocker is the same each day: the `daily-improve` GitHub App
  token does not carry `workflows: write`, so the bot can prepare
  workflow-file edits but the closing `git push` fails the moment any
  changed file is under `.github/workflows/`. Net result: 17 → 18 → 19
  → 20 days of zero RSS. Open improve PRs already in the queue: **#28**
  (2026-04-29), **#30** (2026-05-01), **#32** (2026-05-11).

**Three ways to break the loop**:

1. **Grant the `daily-improve` App `workflows: write`** (in the GitHub
   App's repository permissions; also bump `permissions:` on the
   workflow). Every subsequent daily PR then self-applies.
2. **Merge one of these PRs manually.** A single human merge backfills
   the fix and resets the clock.
3. **Stop running `daily-improve` until it's unblocked.** Pause the
   cron in `.github/workflows/daily-improve.yml` so the workflow stops
   generating these "same diagnosis again" PRs that pile up unmerged.

If none of the above happens, tomorrow's `improve/2026-05-18` PR will
say the exact same thing.

Three of the eleven configured feeds yield no usable content today. The
workflow's `curl ... 2>/dev/null || echo "<rss></rss>"` pattern only
catches transport failures — it silently passes through HTTP-error
bodies (HTML 404 pages, frozen feeds, etc.) as if they were valid
input. This PR replaces it with shape-validation (`<rss|<rdf|<feed`
sniffing) so wrong-shape responses get stubbed and skipped instead.
