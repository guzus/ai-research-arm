### 3. Self-improvement workflow cannot patch workflow files (META)

Discovered while landing this PR: the `daily-improve.yml` workflow runs
under the default `secrets.GITHUB_TOKEN`, which **does not include the
`workflows` permission scope**. So when this run tried to push commits
that modified `.github/workflows/*.yml`, the push was rejected with:

```
refusing to allow a GitHub App to create or update workflow
.github/workflows/<file> without `workflows` permission
```

This is a structural limitation, not a one-off failure: every prior
self-improvement cycle that touched a workflow file would have hit
this. The workaround used in this PR is to ship the workflow patches as
a `*.patch` file at the repo root that the maintainer applies with
`git am` (or `git apply`). The proper fix is to grant a PAT with
`workflows` scope to the daily-improve workflow.

### `.github/workflows/daily-improve.yml` (in patch)

Added a deterministic **Pipeline freshness audit** step that runs
before the Claude analysis step. It walks each `research/<subdir>`,
finds the newest file's mtime, and emits a per-pipeline status table to
`$GITHUB_STEP_SUMMARY`. Pipelines older than their per-pipeline threshold
are listed in a `stale_pipelines` step output that is surfaced into the
Claude prompt as `STALE_PIPELINES`.

This converts silent staleness into a hard signal the next improvement
cycle can act on (the prompt now has explicit instructions to treat
stale pipelines as higher-priority than coverage additions).

| rss | 3 | hourly |

| twitter-deepseek | 9 | every 6h |
